📝 Fluxo de Gravação (Workflow Principal)

Siga o caminho feliz para gerar seu primeiro projeto pronto para integração contínua (CI/CD).

1

Configurando a Feature e Cenário

Abra o painel. No campo principal, digite o agrupamento lógico (ex: Login Módulo Fiscal). No botão inferior, crie os Cenários um a um (ex: Login com Credenciais Válidas).

2

Gravação Dinâmica (Spy Mode)

Pressione Iniciar Gravação. A borda da extensão ficará vermelha. Navegue no seu software livremente.

Ações Rasteadas: Cliques em botões/links, preenchimento de inputs textuais e numéricos, interações de rádio button/checkbox e seleções de <select> nativos.

Dica Crítica: Para valiações de tela, use o "Inspecionar Elemento" 👁️ (na barra inferior do painel). Clique em um botão que prova que a tela carregou e escolha "Validar que o elemento existe". É crucial para estabilizar testes assíncronos.
3

Edição Fina e Validação Visual

Pare a gravação. Você verá os cards com cada Step (Gherkin). Clicando no card, ele expande.

  • Drag and Drop: Clique e arraste para reordenar passos (ou deletá-los no 🗑️).
  • Spotlight: Dúvida se o XPath gerado está apontando certo? Clique no olho (👁️) do lado do campo Custom XPath. O elemento piscará na tela principal com uma aura verde fluorescente se ele for detectado com sucesso.

⚡ Auditoria de Performance (Lighthouse)

Não basta funcionar, tem que ser rápido! Selecione passos estratégicos marcando com (⚡) e defina uma Nota Mínima. O gerador injeta código que avalia Core Web Vitals (LCP, CLS, etc). Se a nota ficar abaixo do esperado, o teste de automação falha, garantindo a velocidade da sua página em cada deploy.

Como funciona o cálculo da Nota de Performance?

A Auditoria de Performance usa o motor open-source do Google Lighthouse por trás dos panos. Quando você exporta o projeto e executa o teste (seja no Cypress, Playwright ou Python), o navegador abre a página em modo "Headless" e simula o carregamento sob condições controladas (Throttle de rede e CPU).

A "Nota de Performance" (que vai de 0 a 100) é calculada com base em métricas críticas chamadas Core Web Vitals, que medem a experiência real do usuário. As principais são:

  • LCP (Largest Contentful Paint): Mede o tempo até o maior elemento de texto ou imagem ficar visível na tela. Um bom LCP é menor que 2.5 segundos.
  • CLS (Cumulative Layout Shift): Mede a estabilidade visual. Se a página "pula" enquanto carrega devido a imagens ou anúncios sem tamanho definido, a nota cai.
  • TBT (Total Blocking Time): Mede quanto tempo a página ficou "travada" (sem poder ser clicada pela thread principal estar ocupada executando JavaScript pesado).
  • FCP (First Contentful Paint): O tempo até o primeiro pixel de conteúdo (texto, imagem, canvas) ser pintado na tela.
  • SI (Speed Index): Quão rápido o conteúdo da página é visualmente preenchido durante o carregamento.
💡 Como o teste falha? Se você definiu a Nota Mínima como 90 na Extensão, mas o Lighthouse calculou que a página tem nota 85 (devido a uma imagem muito pesada atrasando o LCP), o próprio framework de automação lançará um erro do tipo:

AssertionError: Performance score 85 is below the threshold 90

Isso trava o seu pipeline de CI/CD (Github Actions, Jenkins, etc) impedindo que uma tela excessivamente lenta chegue até o usuário final em produção!

⚙️ Motor de Exportação (Arquitetura POM)

Nós não exportamos "scripts sujos". O painel lê suas marcações virtuais e "Decompila" o HTML criando três camadas rigorosas (MVC dos Testes):

  1. Features: Toda a inteligência de negócios declarada em sintaxe Humana (Cucumber).
  2. Steps Definitions: As expressões que colam o Idioma humano aos componentes.
  3. Page Objects (O Segredo Mágico): As classes com atributos Orientados a Objetos (Encapsulamento). O locator do elemento não fica perdido, fica fixado como propriedade da Classe na pasta pages/.

Selecione um cenário, clique em Exportar Projeto Completo (.zip). O arquivo já vem com arquivos package.json / requirements.txt mapeados. Você só precisa descompactar e dar Play.

🔧 Troubleshooting Crítico (FAQ)

1. Elementos SPA's, modais não finalizam a gravação?

A gravação tenta disparar eventos de 'Change' e 'Blur' em inputs para simular o onChange do React de verdade. Às vezes o seu componente de calendário ignora. Nessas horas, apenas dê um "Clique" puro fora da tabela ou em uma área branca da DOM. Isso ativará o evento universal da janela focada e forçará salvar o Step na interface. Você pode apagar esse clique depois no 🗑️.

2. O seletor quebrou de novo! Porque?

A web não dorme. Se a sua empresa usa bibliotecas como *Styled Components* (ex: class="sc-jTk1x kZlYxA"), CSS nunca vai funcionar. Expanda o passo e altere a prioridade para o rádio button que mostra [XPath contendo Nome do Elemento (Aria-label/Text)]. O seletor buscará o elemento pelo Título sem depender da estrutura CSS que os frameworks de Front mudam toda hora.

3. O ZIP está gerando com erro de dependência.

Lembre-se: O fato de você baixar não instala automaticamente os binários dos navegadores nativos instalados no seu Core do Windows. Para Playwright rodar a primeira vez na máquina é mandatório o pacote do Chromium e Webkit ser instalado previamente invocando: npx playwright install antes de bater um npm test.
